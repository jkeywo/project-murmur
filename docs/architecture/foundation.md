# Project Murmur foundation

Project Murmur is a turn-based ASCII social-stealth roguelike. This document
records the agreed MVP boundaries. It is intentionally a design foundation, not
an implementation plan or gameplay implementation.

## MVP world

The MVP generates a two-storey nightclub from a room graph, then connects it
with corridors and populates it. The first building includes an entrance,
dance floor, bar, VIP lounge, kitchen, storage, manager and security offices,
bathrooms, maintenance, and loading bay. It has public, VIP, staff, and private
access zones.

Generation is room-first, not tile-first. Each room records access level,
lighting, exits, cover objects, containers, and routine waypoints before tiles
and corridors are placed. The initial fixed NPC roles are civilians; bartender,
cleaner, technician, and manager staff; guards; and a target drawn from the
staff model. Civilians can wander, drink, dance, panic, and flee; guards can
patrol, investigate, escort, arrest, and fight. All NPCs hold routine, current
state, memory, knowledge, facing, and alertness.

The generated level supplies the target, guards, staff, civilians, access data,
containers, routes, and a terse factual mission briefing. The briefing reports
facts generated by the level rather than flavour text or hand-authored story.

Every required room is physically reachable from every other room. Generation
must also prove dependency-safe access: any required key source is reachable
without that key, and any required disguise is obtainable without already
wearing it, from an actor's schedule or idle position. A wardrobe may be added
when necessary to make a disguise available or avoid placing all copies in high
traffic. Areas that have no valid disguise are exempt from this rule.

## Simulation model

The runtime is an authoritative discrete-turn simulation. Rooms, access,
actors, objects, containers, mission facts, alertness, routines, and knowledge
belong to the world model. NPCs use routines and state machines; the target
uses the same base model as staff with a richer schedule.

NPC perception is line of sight plus facing cone and range. The player has no
facing. NPC facing carries no permanent map marker (Round One decision:
single-cell tiles); inspecting a visible NPC — mouse hover or the look
cursor — overlays every tile their perception currently covers. Disguises define allowed zones, suspicious observers, and illegal
equipment. Initial planned actions are movement, wait, look, garrote, shoot,
carry, drop or hide a body, change disguise, and open or close a door.

The player has a worn-disguise slot, carried-body state, two hands, and six
general inventory slots. Keys and invitations are ordinary inventory items.
Civilian VIP access checks for a persistent, non-consumable invitation item;
tagged VIP NPCs carry invitations that can be looted or pickpocketed.

The MVP permission matrix is explicit: civilian clothing accesses public areas
and VIP only with an invitation; staff clothing accesses public and staff areas
plus authored partial private access; guard and manager clothing access all
zones. Guards detect illegal access, visible weapons, bodies, unconscious
people, and gunshots; suspicion grows over time. Blood and missing-target
detection are deliberately deferred.

Relaxed NPCs act on staggered alternating turns. Suspicious, investigating,
alerted, escorting, fleeing, and combat NPCs act every turn. Suspicion and
alert propagate only through line of sight: a suspicious guard affects guards;
an alert guard affects guards and civilians. Recipients know only the source or
incident location and cannot react until their next action-selection step.

## Controller-neutral actions

The player controller, AI, replay, and tests all submit the same primitive
`ActionIntent` into a prepared action slot for a specific turn. Once the turn's
batch is frozen, resolution must not branch on its source.

After resolving turn N, AI controllers prepare actions for N+1. Immediately
before N+1 begins, the player controller translates one command into its action
for that same turn and freezes it with the already-prepared AI actions.

## Player command queue

The player controller owns a bounded FIFO queue of commands. A queue entry is
an intention, never a prevalidated action. Its head is translated only when it
is due to execute, using current world state and stable domain IDs for targets.
Input is polled between cooperative simulation batches and is appended without
silently dropping existing input. Capacity must be visible; overflow rejects
new input visibly.

Valid commands produce an action and resolve exactly one turn. A command that
cannot be submitted from the pre-turn state is rejected: it consumes no turn,
does not mutate state or consume randomness, and leaves prepared AI actions for
that upcoming turn unchanged. If the rejected command came from a queue, remove
it and cancel all remaining queued commands with a concise reason.

An action that was valid when submitted may still fail in simultaneous
resolution; that turn still passes. The default queue continuation policy is to
cancel remaining commands after a player action resolves as failed or
interrupted, while completed or partially completed actions continue.

The simulation never tries to infer tactical significance or automatically
protect the player from detection, alarms, changed line of sight, or combat.
Queue control is explicit: clear, remove newest command, and the internal
look-mode pause. Leaving look mode resumes consumption immediately (Round One
playtest: an implementation that stayed paused after exit stranded the player
in an unplayable state).

The queue is an architectural mechanism, not a player-facing one (Round One
decision): it has no HUD or start-screen surface and no manual pause toggle.
Escape clears the queue and Backspace removes its newest command. The queue
capacity is exactly 32 commands; input beyond it is rejected with a log line
without disturbing queued input.
Targeted commands
store stable actor or object IDs, never runtime entity IDs, and resolve those
IDs only at execution time. A future controller-level `TravelTo` command may
derive a current-state movement action one turn at a time; it is not a
simulation action itself.

The controller may later compact repeated queued commands into a command plus a
remaining count, but this must be behaviourally identical to individual queued
commands. In look mode, movement controls a detached inspection cursor instead
of enqueueing player movement.

## Resolution, stealth, and outcomes

An invalid command cannot be submitted from the pre-turn state. It is rejected
without advancing time, mutation, or randomness; prepared AI actions stay fixed
for that same upcoming turn. If it was queued, remove it and cancel its
remainder. A valid action can still fail while the simultaneous batch resolves;
time passes in that case and remaining queued commands cancel. Completed or
partially completed actions continue the queue.

Garrote requires adjacency behind the target, both hands free, and no carried
body. The drawn silenced pistol is a visible illegal item with six generated
rounds; it kills a visible in-range target and creates a local sound incident.
Alert guards pursue and attempt arrest. Arrest ends the mission; lethal guard
combat begins only after player violence or arrest resistance.

Crouch is a toggleable player state, not body hiding. It permits movement,
lowers the player's line of sight, and blocks sight through low cover such as
desks. Being seen crouching is inherently suspicious.

Bodies are visible evidence on their tile. Carrying one occupies both hands,
halves movement cadence, and prevents garrote and pistol actions. Each
container holds one body and removes it from ordinary sight when hidden.

The mission succeeds only after the generated target is dead and the player
leaves through any generated external exit, at minimum entrance or loading bay.
It fails only when the player dies or a guard arrests them irreversibly.
Detection, alarm, and combat are playable states.

The primitive action catalogue is move, wait, look, garrote, shoot, carry body,
drop body, hide body, change disguise, open door, and close door. Routines are
generated from the room graph and role waypoints—moving, waiting, interacting,
and repeating—rather than hand-authored encounter scripts.

The generated briefing reports target identity and likely locations, the reason
for elimination, security counts, available disguises, restricted areas, known
container counts, and valid extraction exits.

## Determinism

The mission seed determines layout, population, schedules, items, and all
tie-breaker randomness. Replaying accepted player commands from that seed must
reproduce the same turn-by-turn result.

## Presentation and speed

Simulation has no artificial turn delay, but processes a bounded cooperative
batch before polling input and redrawing. Instant, every-N-turn, every-turn,
and delayed rendering modes must yield identical simulation results.

Before a mission, a splash/start screen gives a brief explanation of the
social-stealth objective and failure conditions, lists active key bindings, and
offers a start action. When a mission ends, a game-over screen states the exact
reason—successful extraction, death, or arrest—and provides a return-to-start
screen action. These are interface modes: they do not advance simulation time.

## Delivery targets

The eventual game has two presentation targets over the same simulation
contract:

- A native Bevy executable with console-oriented presentation for local and
  desktop play.
- A WebAssembly build hosted as a static GitHub Pages site.

Continuous integration runs on each push. When required checks succeed on the
default branch, CI builds the web artifact and deploys it to GitHub Pages;
failed checks cannot deploy. This is a delivery requirement only—no workflow or
runtime implementation is added by this foundation.

## Planned vertical-slice order

1. Tile engine and FOV.
2. Procedural nightclub generation.
3. Zone and access rules.
4. NPC movement and generated routines.
5. Vision cones and facing.
6. Suspicion and alert propagation.
7. Target generation and disguises.
8. Body handling, garrote, and pistol.
9. Generated briefing and mission outcomes.
